| **01-common** — ID strategy `Long` + `GenerationType.IDENTITY` on `BaseEntity` | Single-tenant / single-Postgres modular monolith rules out the distributed scenarios where UUID PKs win. The one real UUID benefit (non-enumerable public IDs) concerns exactly one outward-facing entity (`Order`) and is solved at `07` with a separate UUIDv7 public reference *alongside* the `Long` PK on that entity only — not engine-wide. Postgres `generated always as identity` maps to Hibernate `IDENTITY`. Keep-it-simple + extract-don't-predict. |
| **01-common** — optimistic locking + timestamp auditing on `BaseEntity` | `@Version` (cheap, prevents lost updates) on every entity via the base class. `created_at`/`updated_at` via Spring Data JPA auditing (`@CreatedDate`/`@LastModifiedDate` + `AuditingEntityListener`, `@EnableJpaAuditing`). User audit (`created_by`/`updated_by`) deferred to `02-auth` — no principal exists yet; `SystemAuditorAware` (returns `system`) is wired now as the seam so the columns drop in without re-plumbing. |
| **01-common** — migrations via Flyway (SQL-first), not Liquibase | Postgres is nailed down; use its specifics rather than abstract the DB. Liquibase's changeset DB-abstraction is wasted on a single-DB engine; Flyway runs raw SQL directly. `V1__baseline.sql` creates the `setting` table; `ddl-auto=validate` in tests proves entity↔migration parity. |
| **01-common** — Spring Boot 4 extracted Flyway auto-config into `spring-boot-flyway` | Empirically discovered: with only `flyway-core` on the classpath, migrations never ran before Hibernate validate (`SchemaManagementException: missing table [setting]`). In Spring Boot 4's modular restructure `FlywayAutoConfiguration` moved out of `spring-boot-autoconfigure` into a standalone `spring-boot-flyway` module (BOM-managed, verified June 2026) that must be depended on explicitly. Adding it restored the auto-run + `entityManagerFactory` depends-on ordering. |
| **01-common** — secret storage = AES-256-GCM, direct, `v1:`-prefixed | Storage format `v1:Base64(IV[12] ‖ ciphertext ‖ tag[16])` — one self-describing string, one column. 96-bit IV freshly random per encryption via `SecureRandom` (NIST SP 800-38D: never reuse an IV under one key — https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf); 128-bit GCM tag appended by JCA `AES/GCM/NoPadding`. The `v1:` prefix enables future key/algorithm rotation (recognise + reject older payloads). Decryption is an explicit `CryptoService.decrypt()` call at the use site, not a transparent JPA converter — keeps plaintext out of entity memory except when needed. Tamper-detection is a unit test: a mutated byte fails authentication (throws), never returns wrong plaintext. Sources: NIST SP 800-38D, OWASP Cryptographic Storage Cheat Sheet (https://cheatsheetseries.owasp.org/cheatsheets/Cryptographic_Storage_Cheat_Sheet.html), Java JCA. |
| **01-common** — master key from env, generated by a CLI | Key read from `nobilis.crypto.master-key` (env-backed placeholder, NEVER committed; `*-local.properties`/`.env` gitignored). `CryptoService` fails fast with guidance if it is absent or not a Base64 256-bit value. `CryptoKeyGenerator` is a plain `main()` (no Spring context, so it cannot leak into app startup) that prints a fresh Base64 key for the operator. |
| **01-common** — repository layer = plain `JpaRepository`, no `BaseRepository` (task-3 decision) | No shared repository method is needed yet; a `@NoRepositoryBean BaseRepository<T extends BaseEntity>` would be speculative abstraction (Simplicity First / extract-don't-predict). Introduce it only when a genuinely shared query/method appears. |
| **01-common** — i18n default RU, fallback chain `requested → RU` | Spring `ReloadableResourceBundleMessageSource` over `classpath:messages` for backend static strings (RU/RO bundles scaffolded now, populated at `05-i18n-static`). `LocaleResolver` reads the `?locale=` query param (back↔front contract; chosen over `Accept-Language` for explicitness/manual testing), accepts `ru`/`ro` lowercase, and silently falls back to the `DEFAULT_LOCALE` constant (`ru`) on absent/blank/unsupported — bad input is never an error. Implemented as a framework-agnostic component taking a `String` (not a Servlet `LocaleResolver`) so `common` stays free of any web dependency; the web modules bind the param at `02`/`03`. |
| **01-common** — Testcontainers 2.0 renamed every module artifact | Empirically discovered: Spring Boot 4.1 manages `org.testcontainers:testcontainers-bom` **2.0.5** (a major bump), in which the module coordinates gained a `testcontainers-` prefix — `org.testcontainers:junit-jupiter` → `testcontainers-junit-jupiter`, `…:postgresql` → `testcontainers-postgresql`. The old names are unmanaged → "version is missing" at POM read. TC 2.0 also moved the container class to a new package and made it non-generic: `org.testcontainers.containers.PostgreSQLContainer<?>` (diamond) is deprecated in favour of `org.testcontainers.postgresql.PostgreSQLContainer` (raw, self-bounded) — caught via an IDE deprecation inspection, not the build. The Flyway/JPA slice (`@SpringBootTest` + `@ServiceConnection` against a real `postgres:18` container) uses the new coordinates and class. |

| **01-common** — Lombok 1.18.46 (`provided`), applied judiciously | Compile-time boilerplate reduction. Version verified against Maven Central (`maven-metadata.xml` `<release>` = 1.18.46, published 2026-04-21) — **not guessed**; JDK 25 support landed in 1.18.40 (changelog, Sept 2025), so 1.18.46 covers our Java 25. Pinned in parent `dependencyManagement` at `provided` scope (annotation processor + compile-time API only; nothing ships at runtime), modules opt in by declaring it without a version. Root `lombok.config`: `config.stopBubbling = true` (don't search outside the repo — other projects on the machine carry their own configs) + `lombok.addLombokGeneratedAnnotation = true`. |
| **01-common** — JDK 25 needs the javac internals opened for Lombok | Lombok rewrites the javac AST; JEP 396 strong encapsulation (JDK 16+) blocks that by default. `maven-compiler-plugin` runs with `<fork>true</fork>` and passes `-J--add-opens=jdk.compiler/com.sun.tools.javac.<pkg>=ALL-UNNAMED` for the eleven `com.sun.tools.javac.*` packages (api, code, comp, file, jvm, main, model, parser, processing, tree, util) to the forked javac JVM — verified empirically warning- and error-free on Java 25. Distinct from the existing `.mvn/jvm.config` `--add-exports` (those are for google-java-format/Spotless running in the Maven JVM, a separate concern). `<proc>full</proc>` names annotation processing explicitly, silencing javac's JDK 21+ "processors found on the class path" warning while keeping classpath discovery of Lombok + spring-boot-configuration-processor. |
| **01-common** — Lombok on entities = `@Getter`/`@Setter`, never `@Data` | `@Data` bundles `@EqualsAndHashCode`/`@ToString` over **all** fields, which on a JPA entity drags lazy associations into equality/`toString` (LazyInitializationException, N+1, broken Set semantics). Instead: `BaseEntity` is `@Getter`-only (id db-generated, version provider-managed, timestamps audited — no setters). `Setting` is `@Getter` + field-level `@Setter` on mutable fields only (natural key `key` stays immutable), `@NoArgsConstructor(access = PROTECTED)` for Hibernate, and `@EqualsAndHashCode(onlyExplicitlyIncluded = true, callSuper = false)` including only the business key — stable across the lifecycle, never touches an association. Services use `@RequiredArgsConstructor` (constructor injection of `final` collaborators) + `@Slf4j`; `CryptoService` keeps its hand-written constructor because it validates/decodes the key (not pure injection). Records (`CryptoProperties`) stay records — immutable value carriers, not converted. Spotless/Checkstyle operate on source and never see Lombok's generated members, so there is no conflict. |

| **01-common** — package-by-**feature**, not by layer | Top-level packages in `common` are capabilities (`crypto`, `settings`, `i18n`, `persistence`), never layer buckets (`config`, `util`, `entity`, `service`). The `config` bucket was dissolved — `I18nConfig` → `i18n`, JPA auditing config + `SystemAuditorAware` → `persistence` — and `entity` renamed to `persistence` so `BaseEntity` sits with the auditing wiring it serves. Layers (controllers, etc.) appear *inside* a feature only once it grows (03+), not as a root split. The decisive payoff is encapsulation: a feature package lets collaborators be **package-private**, so only the intended entry points are `public` — applied here by demoting `GcmCipher` to package-private (only `CryptoService` is the public face of `crypto`; recon confirmed no caller outside the package). Sources (independent, clean-room): javapractices.com "Package by feature, not layer"; ProAndroidDev "Package by Layered Feature" (feature-first, layers nested once justified); Expedia Group and Sahibinden engineering write-ups on using package-private visibility to enforce module boundaries. Crypto is the reference feature: one public service, internals hidden. |

| **01-common** — no crypto key in any committed file, tests included | The master key never lands in a tracked file (the production contract, and what `CryptoService`'s Javadoc promises) — this extends to test resources. `application.properties` carries no key; the integration slice generates a fresh AES-256 key in-memory via `CryptoKeyGenerator.generateBase64Key()` and feeds it to the context with `@DynamicPropertySource` (`nobilis.crypto.master-key`). So a test run uses a real, random key with no value sitting on disk. (No key value is recorded here either.) |

| **01-common** — secret-scan gate = gitleaks (pre-commit + CI) | Discipline is not a control — a real AES master key was nearly committed in `application.properties`, contradicting our own Javadoc. A machine barrier now enforces "no secret value in any committed file": a local `.githooks/pre-commit` runs `gitleaks git --staged` (fail-closed: no gitleaks → no commit) and a required GitHub Actions job (`secret-scan.yml`) runs `gitleaks dir .`, both reading `.gitleaks.toml` (gitleaks 8.30.1, pinned). The config extends gitleaks' default ruleset and adds a rule for a Base64 256-bit value assigned to a secret-ish property (the master-key shape), with an allowlist for `*.example` samples, `${ENV}` placeholders, and dependency/build dirs (`node_modules`/`dist`/`target`). Demonstrated both ways: a planted key blocks the commit (HEAD unchanged), the clean tree passes. One-time per clone: `git config core.hooksPath .githooks`. Source: gitleaks — https://github.com/gitleaks/gitleaks. |
| **02-auth** — opt-in mechanism = feature-flag auto-configuration (resolves the milestone-02 TODO) | A capability must be off until the host explicitly enables it (CLAUDE.md "Boundary: engine vs domain" — opt-in by default, no blanket component-scan; this is the engine boundary rule, *not* numbered principle 4, which is i18n). Chosen mechanism: `@AutoConfiguration` classes registered in `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (the Spring Boot 4 path — `spring.factories` autoconfig was removed in Boot 3; confirmed via context7 against Boot 4.1 docs). The admin-login controller+service are gated with `@ConditionalOnProperty(nobilis.auth.admin-login.enabled=true)` so they mount only when the flag is on; with it absent/false the whole `@AutoConfiguration` is skipped and neither bean exists — proven by an `ApplicationContextRunner` test (`AdminLoginAutoConfigurationTest`). This also solves the sibling-package gap recon flagged (a bare `@SpringBootApplication` component-scan never reaches `engine.auth.*`): `.imports` discovery is classpath-driven, not scan-driven, so the host opts in purely by depending on the `auth` jar. Token primitives split by always-on need: `PasswordHasher` unconditional; `JwtService` gated on a configured `nobilis.auth.jwt.secret` (useless without a key, and avoids a fail-fast boot crash when auth is on the classpath but unconfigured). NOTE — `common` is still unmounted into `app`/`admin` via the same sibling-package gap; this pass deliberately did NOT rewire it (out of scope; admin login does not need common's beans). Source: Spring Boot 4.1 reference, "Creating Your Own Auto-configuration" (context7). |
| **02-auth** — JWT hand-rolled HS256 (JCA), BCrypt via Spring Security crypto | JWT (RFC 7519 / 7515, HS256) implemented directly on `javax.crypto.Mac` + Base64URL + Jackson, mirroring the engine's hand-rolled-primitive house style (`common.crypto.GcmCipher`) and keeping the dependency surface minimal — no `jjwt`/Nimbus. Signature verification uses constant-time `MessageDigest.isEqual`; `exp` is enforced against an injected `Clock` (fixed in tests). Password hashing is NOT hand-rolled — BCrypt is delegated to `org.springframework.security:spring-security-crypto`'s `BCryptPasswordEncoder` (the vetted KDF; only a thin `PasswordHasher` façade is exposed). `auth` gains `spring-webmvc` (MVC API, no embedded server — the lib contributes controllers; the runnable `admin` brings the server) and `jackson-databind`; all versions managed by the Boot BOM. Admin config is `.properties` (not YAML, per project preference): `admin/application.properties` ships `nobilis.auth.admin-login.enabled=false` (safe opt-in default) and documents the three env-supplied secrets (email, BCrypt hash, JWT key) — no secret value committed; tests generate keys/hashes at runtime. Sources: RFC 7519/7515, OWASP Password Storage Cheat Sheet, Spring Security crypto docs. |
