| **01-common** — secret storage = AES-256-GCM, direct, `v1:`-prefixed | Storage format `v1:Base64(IV[12] ‖ ciphertext ‖ tag[16])` — one self-describing string, one column. 96-bit IV freshly random per encryption via `SecureRandom` (NIST SP 800-38D: never reuse an IV under one key — https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf); 128-bit GCM tag appended by JCA `AES/GCM/NoPadding`. The `v1:` prefix enables future key/algorithm rotation (recognise + reject older payloads). Decryption is an explicit `CryptoService.decrypt()` call at the use site, not a transparent JPA converter — keeps plaintext out of entity memory except when needed. Tamper-detection is a unit test: a mutated byte fails authentication (throws), never returns wrong plaintext. Sources: NIST SP 800-38D, OWASP Cryptographic Storage Cheat Sheet (https://cheatsheetseries.owasp.org/cheatsheets/Cryptographic_Storage_Cheat_Sheet.html), Java JCA. |

| **01-common** — JDK 25 needs the javac internals opened for Lombok | Lombok rewrites the javac AST; JEP 396 strong encapsulation (JDK 16+) blocks that by default. `maven-compiler-plugin` runs with `<fork>true</fork>` and passes `-J--add-opens=jdk.compiler/com.sun.tools.javac.<pkg>=ALL-UNNAMED` for the eleven `com.sun.tools.javac.*` packages (api, code, comp, file, jvm, main, model, parser, processing, tree, util) to the forked javac JVM — verified empirically warning- and error-free on Java 25. Distinct from the existing `.mvn/jvm.config` `--add-exports` (those are for google-java-format/Spotless running in the Maven JVM, a separate concern). `<proc>full</proc>` names annotation processing explicitly, silencing javac's JDK 21+ "processors found on the class path" warning while keeping classpath discovery of Lombok + spring-boot-configuration-processor. |
| **01-common** — Lombok on entities = `@Getter`/`@Setter`, never `@Data` | `@Data` bundles `@EqualsAndHashCode`/`@ToString` over **all** fields, which on a JPA entity drags lazy associations into equality/`toString` (LazyInitializationException, N+1, broken Set semantics). Instead: `BaseEntity` is `@Getter`-only (id db-generated, version provider-managed, timestamps audited — no setters). `Setting` is `@Getter` + field-level `@Setter` on mutable fields only (natural key `key` stays immutable), `@NoArgsConstructor(access = PROTECTED)` for Hibernate, and `@EqualsAndHashCode(onlyExplicitlyIncluded = true, callSuper = false)` including only the business key — stable across the lifecycle, never touches an association. Services use `@RequiredArgsConstructor` (constructor injection of `final` collaborators) + `@Slf4j`; `CryptoService` keeps its hand-written constructor because it validates/decodes the key (not pure injection). Records (`CryptoProperties`) stay records — immutable value carriers, not converted. Spotless/Checkstyle operate on source and never see Lombok's generated members, so there is no conflict. |

| **01-common** — package-by-**feature**, not by layer | Top-level packages in `common` are capabilities (`crypto`, `settings`, `i18n`, `persistence`), never layer buckets (`config`, `util`, `entity`, `service`). The `config` bucket was dissolved — `I18nConfig` → `i18n`, JPA auditing config + `SystemAuditorAware` → `persistence` — and `entity` renamed to `persistence` so `BaseEntity` sits with the auditing wiring it serves. Layers (controllers, etc.) appear *inside* a feature only once it grows (03+), not as a root split. The decisive payoff is encapsulation: a feature package lets collaborators be **package-private**, so only the intended entry points are `public` — applied here by demoting `GcmCipher` to package-private (only `CryptoService` is the public face of `crypto`; recon confirmed no caller outside the package). Sources (independent, clean-room): javapractices.com "Package by feature, not layer"; ProAndroidDev "Package by Layered Feature" (feature-first, layers nested once justified); Expedia Group and Sahibinden engineering write-ups on using package-private visibility to enforce module boundaries. Crypto is the reference feature: one public service, internals hidden. |

| **01-common** — no crypto key in any committed file, tests included | The master key never lands in a tracked file (the production contract, and what `CryptoService`'s Javadoc promises) — this extends to test resources. `application.properties` carries no key; the integration slice generates a fresh AES-256 key in-memory via `CryptoKeyGenerator.generateBase64Key()` and feeds it to the context with `@DynamicPropertySource` (`nobilis.crypto.master-key`). So a test run uses a real, random key with no value sitting on disk. (No key value is recorded here either.) |

Source: gitleaks — https://github.com/gitleaks/gitleaks. |
| **02-auth** — opt-in mechanism = feature-flag auto-configuration (resolves the milestone-02 TODO) | A capability must be off until the host explicitly enables it (CLAUDE.md "Boundary: engine vs domain" — opt-in by default, no blanket component-scan; this is the engine boundary rule, *not* numbered principle 4, which is i18n). Chosen mechanism: `@AutoConfiguration` classes registered in `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (the Spring Boot 4 path — `spring.factories` autoconfig was removed in Boot 3; confirmed via context7 against Boot 4.1 docs). The admin-login controller+service are gated with `@ConditionalOnProperty(nobilis.auth.admin-login.enabled=true)` so they mount only when the flag is on; with it absent/false the whole `@AutoConfiguration` is skipped and neither bean exists — proven by an `ApplicationContextRunner` test (`AdminLoginAutoConfigurationTest`). This also solves the sibling-package gap recon flagged (a bare `@SpringBootApplication` component-scan never reaches `engine.auth.*`): `.imports` discovery is classpath-driven, not scan-driven, so the host opts in purely by depending on the `auth` jar. Token primitives split by always-on need: `PasswordHasher` unconditional; `JwtService` gated on a configured `nobilis.auth.jwt.secret` (useless without a key, and avoids a fail-fast boot crash when auth is on the classpath but unconfigured). NOTE — `common` is still unmounted into `app`/`admin` via the same sibling-package gap; this pass deliberately did NOT rewire it (out of scope; admin login does not need common's beans). Source: Spring Boot 4.1 reference, "Creating Your Own Auto-configuration" (context7). |
| **02-auth** — JWT hand-rolled HS256 (JCA), BCrypt via Spring Security crypto | JWT (RFC 7519 / 7515, HS256) implemented directly on `javax.crypto.Mac` + Base64URL + Jackson, mirroring the engine's hand-rolled-primitive house style (`common.crypto.GcmCipher`) and keeping the dependency surface minimal — no `jjwt`/Nimbus. Signature verification uses constant-time `MessageDigest.isEqual`; `exp` is enforced against an injected `Clock` (fixed in tests). Password hashing is NOT hand-rolled — BCrypt is delegated to `org.springframework.security:spring-security-crypto`'s `BCryptPasswordEncoder` (the vetted KDF; only a thin `PasswordHasher` façade is exposed). `auth` gains `spring-webmvc` (MVC API, no embedded server — the lib contributes controllers; the runnable `admin` brings the server) and `jackson-databind`; all versions managed by the Boot BOM. Admin config is `.properties` (not YAML, per project preference): `admin/application.properties` ships `nobilis.auth.admin-login.enabled=false` (safe opt-in default) and documents the three env-supplied secrets (email, BCrypt hash, JWT key) — no secret value committed; tests generate keys/hashes at runtime. Sources: RFC 7519/7515, OWASP Password Storage Cheat Sheet, Spring Security crypto docs. |
| **02-auth** (B0.1) — account/identity/realm model: value memberships vs. an audited entity | Realm membership is modelled as an `@ElementCollection<Realm>` on `Account` (`Account.java:66-70`), yielding an `account_realm` join with a composite `(account_id, realm)` key and no surrogate — a realm is a *value* the account holds, with no identity or audit of its own, so it is deliberately **not** a join entity. Against that, `AccountIdentity` IS a full audited entity extending `BaseEntity` (`AccountIdentity.java:52`): attaching a way to log in ("linked Telegram on date X") is an audited event with its own lifecycle, so it earns an id + audit columns. That entity-vs-membership fork is the decision. `AccountIdentity` equals by its business key `(providerType, externalId)` via `@EqualsAndHashCode(onlyExplicitlyIncluded = true, callSuper = false)` (`AccountIdentity.java:50`), mirroring how `Setting` keys equality in 01-common — never the generated id (convention). `Account` has no natural business key, so it keeps **JVM identity** (no `@EqualsAndHashCode` at all — `Account.java:59`), since the convention forbids id-based equals. Root entity named `account`, `@Table(name = "account")` (`Account.java:57`), not `user` — `user` is a SQL/Postgres reserved word and "account" is the better semantic fit for an identity root independent of how it authenticates. |
| **02-auth** (B0.2) — role/permission model: entity-join vs. value-collection; string catalog, not enum | `account_role` is a `@ManyToMany @JoinTable` on `Account` (`Account.java:73-77`) → a composite `(account_id, role_id)` key with no surrogate: an entity↔entity join (both sides are entities), the same owner-collection philosophy as `account_realm` expressed for an entity target. `role_permission` is an `@ElementCollection<String>` on `Role` (`Role.java:60-63`), explicitly mirroring `account_realm` — the permissions a role grants are values, not entities. Permissions are plain `String` constants in `EnginePermissions` (`EnginePermissions.java:30`), **NOT a Java enum**: a domain product can ship its own catalog and add authorities without editing or recompiling the engine (Java enums cannot be extended); the DB stores the constant's string. `PermissionResolver` is placed in the `account` package, not `role` (`PermissionResolver.java:16`, `:33`) — the effective-permission walk runs one way `account → role`, so co-locating it with `Account` avoids a package cycle. |
| **02-auth** (B0.3) — thick token + hand-rolled gate filter (a mechanism, not a policy) | The token is **thick**: realms + effective permissions ride inside the signed JWT (the 4-arg `JwtService.issue` — `JwtService.java:106-117`), so a gate can authorize with no per-request DB round-trip. Trade-off accepted: revocation is not instant → to be mitigated by a short TTL later. The gate is a hand-rolled `OncePerRequestFilter` (`JwtAuthenticationFilter.java:39`), **NOT Spring Security**: no new dependency, nothing is force-secured, and it is consistent with the hand-rolled-on-JCA line (`JwtService`, `common.crypto`). It is a pure MECHANISM — a missing or invalid token simply proceeds unauthenticated and the filter never rejects (`JwtAuthenticationFilter.java:35`, `:64`); deciding the admin/client contour is the host's job at milestone 03. The wire stays backward-compatible: the thin 2-arg `issue(subject, roles)` overload still exists (`JwtService.java:101`) and `validate` reads realms/permissions null-tolerantly through `readStringList` (empty when the claim is absent — `JwtService.java:188`), so an old `{sub,roles,iat,exp}` token still validates. The thick token's first issuer is the config-admin, hardcoding `realms=[ADMIN]` + all `EnginePermissions.ALL` with no DB (`AdminLoginService.java:68-72`); *why* that issuer stays DB-free is B0.4. |
| **02-auth** (B0.4) — config-admin is an intentional STATELESS bootstrap path, not migrated to a DB account | `AdminLoginService` authenticates the single config-admin from `AdminLoginProperties` and issues a thick token (realms=`[ADMIN]` + the full `EnginePermissions.ALL`) with **no `account` row** — by design, not a missing migration. Rationale: config-admin is bootstrap/owner access that must keep working even against an empty or broken database (before any account exists, or if account infrastructure is down), so it cannot depend on DB state. Database-backed accounts arrive later with the identity providers (Telegram/SMS/email) and resolve their realms/permissions from their roles via `PermissionResolver`; the config-admin stands **alongside** them, never replaced by them. Locked by `AdminLoginAutoConfigurationTest#configAdminIssuesThickTokenWithoutAnAccountRow` — admin login mints its thick token in an `ApplicationContextRunner` with no JPA/persistence (asserts there is no `AccountRepository` bean), so any future coupling of admin login to a DB account breaks the test. `SystemAuditorAware` stays the `system` stub; the real audit principal is a separate later pass. |
| **03-app-admin-shell** — stateless admin host: DB/JPA/Flyway auto-config excluded | The slice (admin login + contour + a placeholder screen) touches no database, but `common` drags Spring Data JPA + Hibernate + Flyway onto the admin classpath, so a bare host refuses to start without a datasource. `AdminApplication` excludes the four auto-configs (`AdminApplication.java:36-41`) — `DataSourceAutoConfiguration`, `HibernateJpaAutoConfiguration`, `DataJpaRepositoriesAutoConfiguration`, `FlywayAutoConfiguration` — so it boots with no Postgres; DB returns in a later pass. **Doc-drift caught:** Spring Boot 4 moved these out of the pre-4 `org.springframework.boot.autoconfigure.*` into module-split packages `…boot.jdbc.autoconfigure` / `.hibernate.autoconfigure` / `.data.jpa.autoconfigure` / `.flyway.autoconfigure` (`AdminApplication.java:20-23`, verified against the 4.1.0 sources jars via the IDE index, not from memory), and the JPA-repositories class was renamed to `DataJpaRepositoriesAutoConfiguration`. Proven by `AdminApplicationTest` (context boots with no `DataSource` bean). |
| **03-app-admin-shell** — contour policy lands (the host job B0.3 deferred) | The gate is a mechanism that never rejects; the admin *contour* is host policy. `AdminContourFilter` (`AdminContourFilter.java:44`), a host-side `OncePerRequestFilter`, reads the claims the gate bound: the login endpoint + `/error` are open, no claims → **401** (`:59`), a token without the `ADMIN` realm → **403** (`:62-63`). It must run after the gate, so order is made deterministic via explicit `FilterRegistrationBean`s (`ContourSecurityConfiguration.java:55-70`): gate at `LOWEST_PRECEDENCE-10`, contour at `LOWEST_PRECEDENCE` (`:45-47`) — auto-registered `Filter` beans both default to `LOWEST_PRECEDENCE` and would tie; referencing the gate bean from a registration bean also suppresses its own double auto-registration (`ServletContextInitializerBeans`). Guarded on `nobilis.auth.jwt.secret` (`:41`), the gate's own trigger, so a secret-less host is simply unsecured, never force-secured. Proven by `AdminContourFilterTest` (401 / 200 / 403). |
| **03-app-admin-shell** — local dev secrets via a gitignored profile; runnable-owns-config | Real admin secrets (JWT key, admin email, BCrypt hash) live ONLY in a gitignored `application-local.properties` — the `*-local.properties` pattern was already present (`.gitignore:47`) — loaded with `-Dspring-boot.run.profiles=local`. | **03-app-admin-shell** (pass 2) — `common` mounts via auto-configuration (sibling-package gap closed) | Hosts' component-scan roots never reach `engine.common.*`, so common's beans were unavailable in `app`/`admin` (present only in tests that scanned common). common now ships `META-INF/spring/…AutoConfiguration.imports` with three feature auto-configs (mirroring auth): `CryptoAutoConfiguration` — `@ConditionalOnProperty("nobilis.crypto.master-key")` + `@Bean CryptoService` (`CryptoAutoConfiguration.java:32-34`), so a keyless host simply has no crypto (never the fail-fast boot `CryptoService`'s key validation would cause); `I18nAutoConfiguration` — unconditional `@Bean MessageSource` + resolver, `@AutoConfiguration(before = MessageSourceAutoConfiguration.class)` so the engine's UTF-8 source wins and Boot's default backs off, with the resolver bean renamed `engineLocaleResolver` to avoid the Spring-MVC servlet `localeResolver` name clash (`I18nAutoConfiguration.java:37,55`); and `PersistenceAutoConfiguration` (auditing). | **03-app-host-boot** — the `app` host did not boot at all; fixed with a properties-only `spring.autoconfigure.exclude`, NOT a code change | Recon set out to "add a first endpoint" and found the premise wrong: `app` had an empty `src/main/resources`, so nothing declared the exclusions. `common` puts Spring Data JPA + Flyway + the Postgres driver on every host's classpath, so `DataSourceAutoConfiguration` engaged and the built jar died at startup — `Failed to configure a DataSource: 'url' attribute is not specified` (reproduced by running `app-0.1.0-SNAPSHOT.jar`, not inferred). | **03-app-host-boot** — `app` keeps a plain `@SpringBootApplication`; admin's spelled-out `@SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan` is deliberately NOT copied | `AdminApplication`'s elaborate form exists for exactly one reason: a third `excludeFilters` entry keeping its DB-only controllers (`SettingsController`/`RoleController`/`AccountController`, each registered by a conditional `@Bean`) out of the component scan, where they would otherwise fail the stateless boot. `app` has no conditionally-registered controllers, so the same treatment would be cargo-cult — mirroring a mechanism while dropping the constraint that motivated it. | **03-app-host-boot** — first live endpoint is a hand-rolled `GET /api/health` → `{"status":"UP"}`, not a landing page or a content API | Three options were weighed. **Chosen:** a liveness probe — it promises nothing about portal content, so it cannot go stale as the portal grows. **Rejected:** serving the landing page from the backend (the Angular `projects/app` already owns that page; two owners of one surface means the backend copy becomes dead code the moment Angular renders the real one). **Rejected:** a `/api/portal/landing` content endpoint (it would fix a CMS response contract before the CMS exists — one guessed consumer, near-certain rework; the same "infrastructure built speculatively" defect BL-001 defers the extension points for). | **Methodology — reversal** — direct-access boundary narrowed from "by work class" to "`.md` only"; the coding agent is now the SOLE authority on non-.md changes | The prior round of this canon (work-class boundary: docs/tooling-scripts/build-config directly editable, product code only via the coding agent) was tried and hit a real incident: a direct-access edit to a NEW file (`docs/architecture-backlog.md`, created outside the coding agent's tracked git flow) was lost — untracked, dropped by a `git stash` that didn't use `--include-untracked` — and, separately, was invisible to a coding-agent session working from a different checkout of the same repo, which then had to gate a pass on a file the operator believed already existed. Canon section renamed accordingly. |
