| **01-common** — secret storage = AES-256-GCM, direct, `v1:`-prefixed | Storage format `v1:Base64(IV[12] ‖ ciphertext ‖ tag[16])` — one self-describing string, one column. 96-bit IV freshly random per encryption via `SecureRandom` (NIST SP 800-38D: never reuse an IV under one key — https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf); 128-bit GCM tag appended by JCA `AES/GCM/NoPadding`. | **01-common** — Lombok on entities = `@Getter`/`@Setter`, never `@Data` | `@Data` bundles `@EqualsAndHashCode`/`@ToString` over **all** fields, which on a JPA entity drags lazy associations into equality/`toString` (LazyInitializationException, N+1, broken Set semantics). Instead: `BaseEntity` is `@Getter`-only (id db-generated, version provider-managed, timestamps audited — no setters). `Setting` is `@Getter` + field-level `@Setter` on mutable fields only (natural key `key` stays immutable), `@NoArgsConstructor(access = PROTECTED)` for Hibernate, and `@EqualsAndHashCode(onlyExplicitlyIncluded = true, callSuper = false)` including only the business key — stable across the lifecycle, never touches an association. Services use `@RequiredArgsConstructor` (constructor injection of `final` collaborators) + `@Slf4j`; `CryptoService` keeps its hand-written constructor because it validates/decodes the key (not pure injection). Records (`CryptoProperties`) stay records — immutable value carriers, not converted. Spotless/Checkstyle operate on source and never see Lombok's generated members, so there is no conflict. |

| **01-common** — package-by-**feature**, not by layer | Top-level packages in `common` are capabilities (`crypto`, `settings`, `i18n`, `persistence`), never layer buckets (`config`, `util`, `entity`, `service`). The `config` bucket was dissolved — `I18nConfig` → `i18n`, JPA auditing config + `SystemAuditorAware` → `persistence` — and `entity` renamed to `persistence` so `BaseEntity` sits with the auditing wiring it serves. Layers (controllers, etc.) appear *inside* a feature only once it grows (03+), not as a root split. The decisive payoff is encapsulation: a feature package lets collaborators be **package-private**, so only the intended entry points are `public` — applied here by demoting `GcmCipher` to package-private (only `CryptoService` is the public face of `crypto`; recon confirmed no caller outside the package). Sources (independent, clean-room): javapractices.com "Package by feature, not layer"; ProAndroidDev "Package by Layered Feature" (feature-first, layers nested once justified); Expedia Group and Sahibinden engineering write-ups on using package-private visibility to enforce module boundaries. Crypto is the reference feature: one public service, internals hidden. |

| **02-auth** — opt-in mechanism = feature-flag auto-configuration (resolves the milestone-02 TODO) | A capability must be off until the host explicitly enables it (CLAUDE.md "Boundary: engine vs domain" — opt-in by default, no blanket component-scan; this is the engine boundary rule, *not* numbered principle 4, which is i18n). Chosen mechanism: `@AutoConfiguration` classes registered in `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (the Spring Boot 4 path — `spring.factories` autoconfig was removed in Boot 3; confirmed via context7 against Boot 4.1 docs). The admin-login controller+service are gated with `@ConditionalOnProperty(nobilis.auth.admin-login.enabled=true)` so they mount only when the flag is on; with it absent/false the whole `@AutoConfiguration` is skipped and neither bean exists — proven by an `ApplicationContextRunner` test (`AdminLoginAutoConfigurationTest`). Token primitives split by always-on need: `PasswordHasher` unconditional; `JwtService` gated on a configured `nobilis.auth.jwt.secret` (useless without a key, and avoids a fail-fast boot crash when auth is on the classpath but unconfigured). | **02-auth** — JWT hand-rolled HS256 (JCA), BCrypt via Spring Security crypto | JWT (RFC 7519 / 7515, HS256) implemented directly on `javax.crypto.Mac` + Base64URL + Jackson, mirroring the engine's hand-rolled-primitive house style (`common.crypto.GcmCipher`) and keeping the dependency surface minimal — no `jjwt`/Nimbus. Signature verification uses constant-time `MessageDigest.isEqual`; `exp` is enforced against an injected `Clock` (fixed in tests). Password hashing is NOT hand-rolled — BCrypt is delegated to `org.springframework.security:spring-security-crypto`'s `BCryptPasswordEncoder` (the vetted KDF; only a thin `PasswordHasher` façade is exposed). `auth` gains `spring-webmvc` (MVC API, no embedded server — the lib contributes controllers; the runnable `admin` brings the server) and `jackson-databind`; all versions managed by the Boot BOM. Admin config is `.properties` (not YAML, per project preference): `admin/application.properties` ships `nobilis.auth.admin-login.enabled=false` (safe opt-in default) and documents the three env-supplied secrets (email, BCrypt hash, JWT key) — no secret value committed; tests generate keys/hashes at runtime. Sources: RFC 7519/7515, OWASP Password Storage Cheat Sheet, Spring Security crypto docs. |
| **02-auth** (B0.1) — account/identity/realm model: value memberships vs. an audited entity | Realm membership is modelled as an `@ElementCollection<Realm>` on `Account` (`Account.java:66-70`), yielding an `account_realm` join with a composite `(account_id, realm)` key and no surrogate — a realm is a *value* the account holds, with no identity or audit of its own, so it is deliberately **not** a join entity. Against that, `AccountIdentity` IS a full audited entity extending `BaseEntity` (`AccountIdentity.java:52`): attaching a way to log in ("linked Telegram on date X") is an audited event with its own lifecycle, so it earns an id + audit columns. That entity-vs-membership fork is the decision. `AccountIdentity` equals by its business key `(providerType, externalId)` via `@EqualsAndHashCode(onlyExplicitlyIncluded = true, callSuper = false)` (`AccountIdentity.java:50`), mirroring how `Setting` keys equality in 01-common — never the generated id (convention). `Account` has no natural business key, so it keeps **JVM identity** (no `@EqualsAndHashCode` at all — `Account.java:59`), since the convention forbids id-based equals. Root entity named `account`, `@Table(name = "account")` (`Account.java:57`), not `user` — `user` is a SQL/Postgres reserved word and "account" is the better semantic fit for an identity root independent of how it authenticates. |
| **02-auth** (B0.2) — role/permission model: entity-join vs. value-collection; string catalog, not enum | `account_role` is a `@ManyToMany @JoinTable` on `Account` (`Account.java:73-77`) → a composite `(account_id, role_id)` key with no surrogate: an entity↔entity join (both sides are entities), the same owner-collection philosophy as `account_realm` expressed for an entity target. `role_permission` is an `@ElementCollection<String>` on `Role` (`Role.java:60-63`), explicitly mirroring `account_realm` — the permissions a role grants are values, not entities. Permissions are plain `String` constants in `EnginePermissions` (`EnginePermissions.java:30`), **NOT a Java enum**: a domain product can ship its own catalog and add authorities without editing or recompiling the engine (Java enums cannot be extended); the DB stores the constant's string. `PermissionResolver` is placed in the `account` package, not `role` (`PermissionResolver.java:16`, `:33`) — the effective-permission walk runs one way `account → role`, so co-locating it with `Account` avoids a package cycle. |
| **02-auth** (B0.3) — thick token + hand-rolled gate filter (a mechanism, not a policy) | The token is **thick**: realms + effective permissions ride inside the signed JWT (the 4-arg `JwtService.issue` — `JwtService.java:106-117`), so a gate can authorize with no per-request DB round-trip. Trade-off accepted: revocation is not instant → to be mitigated by a short TTL later. The gate is a hand-rolled `OncePerRequestFilter` (`JwtAuthenticationFilter.java:39`), **NOT Spring Security**: no new dependency, nothing is force-secured, and it is consistent with the hand-rolled-on-JCA line (`JwtService`, `common.crypto`). It is a pure MECHANISM — a missing or invalid token simply proceeds unauthenticated and the filter never rejects (`JwtAuthenticationFilter.java:35`, `:64`); deciding the admin/client contour is the host's job at milestone 03. The wire stays backward-compatible: the thin 2-arg `issue(subject, roles)` overload still exists (`JwtService.java:101`) and `validate` reads realms/permissions null-tolerantly through `readStringList` (empty when the claim is absent — `JwtService.java:188`), so an old `{sub,roles,iat,exp}` token still validates. The thick token's first issuer is the config-admin, hardcoding `realms=[ADMIN]` + all `EnginePermissions.ALL` with no DB (`AdminLoginService.java:68-72`); *why* that issuer stays DB-free is B0.4. |
| **02-auth** (B0.4) — config-admin is an intentional STATELESS bootstrap path, not migrated to a DB account | `AdminLoginService` authenticates the single config-admin from `AdminLoginProperties` and issues a thick token (realms=`[ADMIN]` + the full `EnginePermissions.ALL`) with **no `account` row** — by design, not a missing migration. Rationale: config-admin is bootstrap/owner access that must keep working even against an empty or broken database (before any account exists, or if account infrastructure is down), so it cannot depend on DB state. Database-backed accounts arrive later with the identity providers (Telegram/SMS/email) and resolve their realms/permissions from their roles via `PermissionResolver`; the config-admin stands **alongside** them, never replaced by them. Locked by `AdminLoginAutoConfigurationTest#configAdminIssuesThickTokenWithoutAnAccountRow` — admin login mints its thick token in an `ApplicationContextRunner` with no JPA/persistence (asserts there is no `AccountRepository` bean), so any future coupling of admin login to a DB account breaks the test. `SystemAuditorAware` stays the `system` stub; the real audit principal is a separate later pass. |
